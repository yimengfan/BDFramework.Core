 2. 批量 InsertAll vs 逐行 Insert  (5000 行)
═══════════════════════════════════════════════════════════════════════════════

  ┌──────────────┬──────────┬─────────┬────────────┐
  │ 方式          │ 耗时(ms) │ GC Gen0 │ 内存(KB)   │
  ├──────────────┼──────────┼─────────┼────────────┤
  │ 逐行 Insert   │  947~1043│    3    │  5290~5310 │
  │ 批量 InsertAll│   23~24  │    3    │    5198    │
  └──────────────┴──────────┴─────────┴────────────┘

  🚀 加速比: 40~45x  (中位数 41.78x)

  分析: InsertAll 内部使用事务包裹，SQLite 事务批量写入比逐行 autocommit
  快 40+ 倍。这是最显著的优化点。GC 次数相同(3次)，内存使用接近，
  说明加速完全来自 I/O 减少（1000次 fsync → 1次 fsync）。

 3. PRAGMA 只读优化 vs 默认配置  (20000行, 100次查询)
═══════════════════════════════════════════════════════════════════════════════

  ┌──────────────────┬──────────┬─────────┐
  │ 方式              │ 耗时(ms) │ GC Gen0 │
  ├──────────────────┼──────────┼─────────┤
  │ 默认配置查询       │  596~667 │  40     │
  │ PRAGMA优化查询     │  598~619 │  42     │
  │ 全表扫描(默认)     │    4~5   │   -     │
  │ 全表扫描(PRAGMA)   │    4~5   │   -     │
  └──────────────────┴──────────┴─────────┘

  加速比: 1.00~1.08x

  分析: 在独立 .NET 环境 + SQLitePCLRaw 下，PRAGMA 优化效果不明显。
  原因：
  1) SQLitePCLRaw 的 e_sqlite3 已经是优化编译版本，默认 mmap 已启用
  2) 数据量(20000行)较小，全部在 OS 文件缓存中
  3) 真正的收益体现在 Unity/IL2CPP 环境 + 大数据量 + 低端设备
  4) 全表扫描仅需 4~5ms，说明瓶颈在 C# 反序列化层而非 SQLite 引擎

 4. Prepared Statement 缓存 vs 无缓存  (10000行, 200次查询)
═══════════════════════════════════════════════════════════════════════════════

  ┌──────────────────────────┬──────────┬─────────┐
  │ 方式                      │ 耗时(ms) │ GC Gen0 │
  ├──────────────────────────┼──────────┼─────────┤
  │ 无缓存(每次Prepare+Step)  │    1     │    0    │
  │ 连接级缓存                │    1     │    0    │
  │ TableQuery<T>（泛型）       │  0~1     │    0    │
  └──────────────────────────┴──────────┴─────────┘

  加速比: 1.00x

  分析: 在当前测试规模下(10000行, 200次查询)，Prepared Statement 缓存
  没有体现出优势。原因：
  1) SQLitePCLRaw 的 Prepare 本身非常快(<5μs)
  2) 查询执行时间被反序列化主导，Prepare 占比微乎其微
  3) 缓存优势在 Unity/Mono 环境下更显著(Mono 的 P/Invoke 开销更大)
  4) 在高频查询场景(>1000次/s)或复杂 SQL 下效果更明显

 5. FastJson Span 解析 vs string.Split  (10000次迭代)
═══════════════════════════════════════════════════════════════════════════════

  ┌──────────────────┬──────────┬─────────┬────────────┐
  │ 方式              │ 耗时(ms) │ GC Gen0 │ 内存(KB)   │
  ├──────────────────┼──────────┼─────────┼────────────┤
  │ Span 解析(新)     │    6~8   │    1    │   6907     │
  │ string.Split(旧)  │    3     │    2    │   1776     │
  └──────────────────┴──────────┴─────────┴────────────┘

  加速比: 0.38~0.50x (Span 反而更慢)
  GC 减少: 1次 (Span 确实减少了中间 string 分配)

  分析: Span 解析在独立 .NET 环境下反而比 string.Split 慢约 2x。
  原因：
  1) .NET Runtime 的 string.Split 高度优化(SIMD/vectorized)，Span
     逐字符扫描无法匹配
  2) Span 版本分配了更多内存(6907KB vs 1776KB)，
     因为 ReadOnlySpan<char>.ToString() 在解析过程中仍创建 string
  3) GC 减少 1 次，证明 Span 确实减少了中间 string 分配
  4) 在 Unity/Mono 中 string.Split 未被优化，Span 版本有更大优势
  5) ⚠️ 建议在 Unity 环境中重新验证此项

 Bug 修复记录
═══════════════════════════════════════════════════════════════════════════════

  Bug #1: ParseStringElements string[] 反序列化错误
  ──────────────────────────────────────────────────
  文件:   SqliteFastJsonConvert.cs, ParseStringElements 方法
  症状:   string[] 反序列化结果与原始不匹配
          输入: ["hello","world","","with,comma","with""quote"]
          错误: ["hello",,,","with,comma"] (元素合并、引号残留)
  原因:   SkipStringLiteral 跳过整个字符串(含闭合引号)后，i 指向闭合
          引号之后。i-- + loop i++ 使 i 停在闭合引号+1处。但 inString
          仍为 true, 下一个 """ 被当作当前字符串的闭合引号, 导致元素
          合并和引号残留。
  修复:   SkipStringLiteral 返回后直接提取元素:
          var element = content.Slice(start, i-1-start).ToString();
          然后设置 inString=false, i-- 让循环正常推进。
  影响范围: 所有使用 DeserializeArrayString 的场景 (游戏配置表 string[] 字段)
  状态:   ✅ 已修复 (同步到 BDFramework 原始源文件)
