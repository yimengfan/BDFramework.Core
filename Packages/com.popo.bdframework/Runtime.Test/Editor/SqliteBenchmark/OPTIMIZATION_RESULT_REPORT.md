2.4 性能回退根因分析
───────────────────────────────────────────────────────────────────────────────

O6-O10 引入的性能回退分布到所有步骤，不仅是 FastSet:
  - Prepare / Step / CreateObj 也都变慢 → 不像是 O6-O10 代码改动直接导致
  - 基准运行环境差异 (batchmode vs Editor GUI, CPU 调度, 缓存状态) 可能是主因

    O1-O5 FastSet=2.36ms → O1-O10 FastSet=3.51ms (+48.7%)
    两遍扫描的第一遍计数遍历虽然消除 List+ToArray, 但总遍历次数从 1 次变 2 次
    对于已缓存的短数组, List.Add + ToArray 实际上可能更快

O6-O10 实测结果 (与 O1-O5 基线对比):

  ⚠️ 回退原因可能包含:

7.2 性能回退分析
───────────────────────────────────────────────────────────────────────────────

回退特征:
  - Prepare +49%~150%, Step +92%~100%, CreateObj +133%~200%
  → 这些步骤未受 O6-O10 改动影响, 全面上扬指向环境/测量差异
  - FastSet +49%~61% → 部分 O6-O10 影响, 部分环境差异
  - ColMap +61%~118% → O8 缓存应有改善但变慢, 指向环境/测量差异

可能的根因 (按可能性排序):
  1. Unity batchmode -nographics 的 CPU 调度不稳定
     (运行时持续输出 GfxDevice 错误, 占用 CPU 资源)
  2. 测量开销: MeasureQuerySteps 的 Stopwatch.Restart()/Stop() per sub-step
     对于亚毫秒级操作, 计时器本身的调度开销可能显著
  3. O6 两遍扫描对短数组 (典型游戏配置 2~10 元素) 不如 List+ToArray:
     List<T>(16) 默认容量足以容纳, 无需扩容; ToArray 对小数组开销极低
     两遍扫描总遍历 2N 次 vs List+ToArray 总遍历 N + N 次, 但后者更缓存友好
  4. O8 ColumnMappingCache.ConcurrentDictionary 查找开销
     对于高频查询, 锁竞争可能抵消缓存收益

⚠️ 需要行动: 在 Unity Editor (非 batchmode) 下重新运行基准测试,
  或在同一运行环境中进行 O1-O5 vs O1-O10 的 A/B 对比测试,
  以排除环境因素, 确认 O6-O10 各优化项的实际效果。

7.5 O6-O10 优化详细说明
───────────────────────────────────────────────────────────────────────────────

O6 — 两遍扫描替代 List+ToArray:
  改动文件: SqliteFastJsonConvert.cs
  改动内容: 所有 6 种 DeserializeArray* 方法改为两遍扫描:
    第一遍: 计数元素 (逗号数+1 或引号对数), 预分配 T[] 数组
    第二遍: 解析并直接填入预分配数组
  消除: List<T>(16) 默认容量分配 + list.ToArray() 二次分配
  单元测试: SqliteFastJsonConvertTest.cs (覆盖空数组、单元素、负数、大数组、空白等)

O7 — float/double Span 零分配解析:
  改动文件: SqliteFastJsonConvert.cs
  改动内容: 新增 ParseFloatFromSpan / ParseDoubleFromSpan:
    手动解析: 符号 → 整数部分 → 小数部分
    特殊值 (NaN/Infinity) 和科学计数法回退到 float.Parse(span.ToString())
  消除: float.Parse(string) 和 double.Parse(string) 的中间 string 分配
  单元测试: SqliteFastJsonConvertTest.cs (覆盖精度、正号、混合整数小数等)

O8 — ColumnMapping 跨查询缓存:
  改动文件: SQLite.cs
  改动内容: 新增 ColumnMappingCache 类 (internal):
    ConcurrentDictionary<(Type, TableMapping, int), CacheEntry>
    CacheEntry 包含 cols[] + fastColumnSetters[]
    ExecuteDeferredQuery 中 ByName 路径: 缓存命中直接复用, 未命中构建后写入缓存
  消除: 每次查询重建 cols[] + fastColumnSetters[] 的开销
  单元测试: 跳过 (internal 类, 正确性由端到端测试隐式保证)

O10 — string[] 内联转义引号:
  改动文件: SqliteFastJsonConvert.cs
  改动内容: 新增 UnescapeString(ReadOnlySpan<char>) helper:
    快速路径: 扫描 "", 无转义引号 → span.ToString() 直接返回 (零额外分配)
    慢速路径: char[] 逐字符复制, "" → " 折叠, new string(buffer, 0, writeIndex)
    DeserializeArrayString 两处调用替换 .ToString().Replace("\"\"", "\"")
  消除: 每个 string 元素的 2 次 string 分配 (ToString + Replace)
  单元测试: SqliteFastJsonConvertTest.cs (O10 专项: 多转义、相邻转义、开头/末尾转义、往返)

⚠️ O6-O10 性能数据已通过 Unity batchmode 基准测试获得 (2025-07-11)
  实测结果显示性能回退, 需进一步 A/B 对比测试排除环境因素
