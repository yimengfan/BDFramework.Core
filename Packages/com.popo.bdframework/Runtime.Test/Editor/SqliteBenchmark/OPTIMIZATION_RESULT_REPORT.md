1. 优化汇总表
═══════════════════════════════════════════════════════════════════════════════

1.1 O1-O5 已验证优化（独立 .NET 基准 + Unity 正确性验证）
───────────────────────────────────────────────────────────────────────────────

| 优化项 | 优化前 | 优化后 | 优化幅度 | 主要手段 |
|--------|--------|--------|---------|---------|
| O1 单次遍历解析 | CountElements + ParseValueElements 双遍历 | 合并为单次遍历 List\<T\> 收集 | 1.3~1.5x（理论） | 消除预扫描计数遍历 |
| O2 int/long Span 零分配 | segment.ToString() + int.Parse(string) 双重分配 | ParseIntFromSpan/ParseLongFromSpan 直接从 Span 解析 | 内存分配大幅降低（Unity/Mono） | ReadOnlySpan\<char\> 零中间 string |
| O3 FastSetter 跳过 Null 检查 | 数组字段每次 ColumnType P/Invoke 判断 Null | 数组字段直接读取，省去 ColumnType 调用 | ReadCol% 从 40~82% 降至 ~0% | CreateArraySetterDelegate 专用委托 |
| O5 List\<T\> 替代 Array.Resize | 预估不准时 Array.Resize 拷贝 | List\<T\>.Add() + ToArray() | ~1.1x | 消除数组拷贝开销 |
| InsertAll 批量导入 | 逐行 Insert autocommit (947~1043ms) | 事务包裹批量写入 (23~24ms) | **41.78x** | 1000次 fsync → 1次 fsync |
| FastJson Span 解析内存 | string.Split: 64,364KB 分配 | Span 解析: 17,592KB 分配 | **-72.7%** | Span 逐字符扫描替代 string.Split |

O3 效果最显著：数组字段 ReadCol 慢路径完全消除，ReadCol% 从 40~82% 降至 ~0%。

1.2 O6-O10 优化（实测回归，待 A/B 对比排除环境因素）
───────────────────────────────────────────────────────────────────────────────

| 优化项 | 优化前 (O1-O5) | 优化后 (O1-O10) | 优化幅度 | 主要手段 |
|--------|---------------|----------------|---------|---------|
| O6 两遍扫描替代 List+ToArray | List\<T\>(16) + ToArray() 单遍 | 第一遍计数预分配数组，第二遍解析填入 | **-56%~108%**（回退⚠️） | 消除 List+ToArray 双重分配，但总遍历次数 1→2 |
| O7 float/double Span 零分配 | float.Parse(string) / double.Parse(string) | ParseFloatFromSpan/ParseDoubleFromSpan 手动解析 | **-56%~108%**（回退⚠️） | 手动解析符号+整数+小数，特殊值回退 |
| O8 ColumnMapping 跨查询缓存 | 每次查询重建 cols[] + fastColumnSetters[] | ConcurrentDictionary 缓存 ColumnMapping | **-56%~108%**（回退⚠️） | 缓存命中直接复用，但 ConcurrentDict 查找有开销 |
| O10 string[] 内联转义引号 | ToString()+Replace() 双重 string 分配 | UnescapeString 快速路径零额外分配 | **-56%~108%**（回退⚠️） | 快速路径 span.ToString()，慢速路径 char[] 折叠 ""→" |

⚠️ O6-O10 实测 E2E 热查询全面回退 56%~108%。回退分布在所有步骤（含未受 O6-O10 影响的步骤），
  疑为 Unity batchmode 环境差异导致。需在同一运行环境中 A/B 对比验证。
  回退幅度按最差表（ScalarOnly）计算，最差 +108%，最优 -56%（HeroSkillParameter 热查询）。

5. O6-O10 性能回退根因分析
═══════════════════════════════════════════════════════════════════════════════

5.1 回退特征
───────────────────────────────────────────────────────────────────────────────

- Prepare +49%~150%, Step +92%~100%, CreateObj +133%~200%
  → 这些步骤未受 O6-O10 改动影响，全面上扬指向环境/测量差异
- FastSet +49%~61% → 部分 O6-O10 影响，部分环境差异
- ColMap +61%~118% → O8 缓存应有改善但变慢，指向环境/测量差异

5.2 可能根因（按可能性排序）
───────────────────────────────────────────────────────────────────────────────

| # | 根因 | 证据 | 可能性 |
|---|------|------|--------|
| 1 | Unity batchmode CPU 调度不稳定 | 运行时持续输出 GfxDevice 错误，占用 CPU 资源；未改动步骤也全面变慢 | 高 |
| 2 | 亚毫秒级 Stopwatch 测量开销 | MeasureQuerySteps 的 Stopwatch.Restart()/Stop() per sub-step 对亚毫秒操作调度开销显著 | 中 |
| 3 | O6 两遍扫描对短数组不如 List+ToArray | List\<T\>(16) 默认容量足以容纳小数组，ToArray 对小数组开销极低；两遍遍历总次数 2N vs List+ToArray N+N 但更缓存友好 | 中 |
| 4 | O8 ConcurrentDictionary 查找开销 | 对于高频查询，锁竞争可能抵消缓存收益；热查询本应命中缓存但 ColMap 仍变慢 | 低~中 |

5.3 需要的验证行动
───────────────────────────────────────────────────────────────────────────────

在 Unity Editor（非 batchmode）下重新运行基准测试，或同一运行环境中 O1-O5 vs O1-O10 A/B 对比，
排除环境因素后确认 O6-O10 各优化项的实际效果。
